Проект: KnockWatch

Система мониторинга детонации двигателя для автопарков
1. Проблема

В автопарках и сервисных центрах сложно отслеживать состояние двигателей большого количества автомобилей.

Основные проблемы:

    диагностика часто проводится слишком поздно

    ранние признаки детонации сложно заметить

    отсутствует централизованное хранение диагностических данных

    нет истории проверок двигателя

    сложно передавать информацию между водителями и механиками

Это может приводить к дорогому ремонту двигателя и простоям транспорта.
2. Решение

KnockWatch — это веб-приложение для мониторинга состояния двигателей.

Система позволяет:

    хранить информацию о машинах автопарка

    загружать диагностические записи работы двигателя

    анализировать звук двигателя

    выявлять подозрение на детонацию

    создавать задачи для механиков

    хранить историю диагностики

В дальнейшем в систему будет интегрирована ML-модель, которая будет автоматически определять наличие детонации по аудиозаписи двигателя.
3. Целевая аудитория
Основная

Автопарки и транспортные компании

Примеры:

    службы доставки

    такси

    логистические компании

    корпоративные автопарки

Дополнительная

Автосервисы и станции технического обслуживания.
4. Основные функции системы
1. Управление автопарком

Система хранит список автомобилей.

Для каждой машины:

    марка

    модель

    VIN

    номер автомобиля

    год выпуска

    пробег

    тип топлива

    статус

Пользователь может:

    добавить машину

    редактировать данные

    просматривать карточку машины

    смотреть историю диагностики.

2. Загрузка диагностической сессии

Пользователь может создать новую диагностическую запись.

Параметры:

    автомобиль

    режим работы двигателя

    аудиозапись двигателя

    комментарий пользователя

После загрузки создаётся диагностическая сессия.
3. Анализ аудиозаписи

Система анализирует загруженный файл.

Результат анализа:

    вероятность детонации

    статус проверки

Возможные статусы:

    normal

    suspicious

    critical

На этапе MVP используется mock-анализ.
В будущем будет подключена ML-модель.
4. Система инцидентов (тикетов)

Если запись выглядит подозрительно, создаётся инцидент.

Инцидент содержит:

    машину

    диагностическую сессию

    уровень критичности

    описание проблемы

    назначенного механика

    статус

Статусы:

    new

    in progress

    resolved

5. Дашборд системы

Главная страница показывает общую статистику.

Примеры метрик:

    количество машин

    активные инциденты

    диагностические проверки за неделю

    критические случаи

Также отображаются:

    последние проверки

    последние инциденты.

6. История диагностики

Для каждой машины хранится история:

    все диагностические сессии

    результаты анализа

    созданные инциденты

    заключения механиков.

5. Пользовательские роли
Администратор

Может:

    управлять пользователями

    добавлять машины

    видеть все данные системы

    просматривать статистику.

Оператор / водитель

Может:

    просматривать машины

    загружать диагностические записи

    создавать новые сессии.

Механик

Может:

    просматривать инциденты

    анализировать записи

    писать заключение

    менять статус задачи.

6. Основные сценарии использования
Сценарий 1 — диагностика

    Оператор выбирает машину

    Загружает запись двигателя

    Добавляет комментарий

    Система создаёт диагностическую сессию

Сценарий 2 — анализ

    Система анализирует запись

    Определяет вероятность детонации

    Назначает статус проверки

Сценарий 3 — работа механика

    Механик видит новый инцидент

    Открывает запись

    проверяет данные

    пишет заключение

    закрывает тикет.
